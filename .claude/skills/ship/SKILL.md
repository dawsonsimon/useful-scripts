# Ship Changes

Automate the full flow: feature branch, commit, and PR — with user approval at each step.

## Step 1: Check current state

- Run `git status` and `git diff --stat` to understand what has changed.
- Run `git branch` to check the current branch.
- If there are no changes (staged or unstaged or untracked), stop and tell the user there is nothing to ship.

## Step 2: Feature branch

- If already on a feature branch (not `main`, `master`, or `develop`), stay on it.
- Otherwise, create a new feature branch:
  - If `$ARGUMENTS` is provided, use it to derive the branch name as `feature/<kebab-case-arguments>`.
  - If no arguments, analyze the changes and generate a short descriptive branch name as `feature/<description>`.
  - Run `git checkout -b <branch-name>`.

## Step 3: Stage changes

- Stage all relevant changes. Exclude files that look like secrets (`.env`, credentials, etc.) and warn the user if any are present.

## Step 4: Commit message approval

- Analyze all staged changes (read diffs, understand the purpose).
- Draft a concise commit message (1-2 lines) that focuses on the "why" not the "what".
- Do NOT include any Co-Authored-By or attribution lines.
- Present the commit message to the user and ask for approval before committing.
- Wait for the user to approve or suggest edits. Do not proceed until approved.

## Step 5: Commit

- Once approved, commit with the approved message.
- Run `npm run lint` after committing to verify code quality. If lint fails, inform the user but do not block.

## Step 6: PR description approval

- Draft a PR title (short, under 70 chars) and body with:
  - `## Summary` — bullet points of what changed
  - `## Test plan` — checklist of testing steps
- Do NOT include any attribution or "Generated by" lines.
- Present the full PR title and body to the user and ask for approval before creating the PR.
- Wait for the user to approve or suggest edits. Do not proceed until approved.

## Step 7: Push and open PR

- Push the branch: `git push -u origin <branch-name>`.
- Determine the correct base branch (`develop` if it exists, otherwise `main` or `master`).
- Create the PR with the approved title and body using `gh pr create`.
- Return the PR URL to the user.
